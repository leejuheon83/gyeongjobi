# 배포 전 체크리스트

## 1. 환경 변수

- [ ] 운영용 Supabase 프로젝트의 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`를 배포 환경(Vercel 등)의 환경 변수로 등록했다 (현재 개발용 프로젝트 키를 그대로 쓰지 않는다)
- [ ] `NEXT_PUBLIC_ATTACHMENT_MAX_FILE_MB` 값을 운영 정책에 맞게 확인했다 (기본 10MB, 스토리지 버킷 하드 리밋도 10MB로 맞춰져 있음 — 늘릴 경우 `supabase/migrations`의 버킷 `file_size_limit`도 함께 변경)
- [ ] `.env.local`이 저장소에 커밋되지 않았는지 확인했다 (`.gitignore`에 이미 포함)
- [ ] 환경 변수가 누락되면 `lib/env.ts`가 원인을 알 수 있는 에러를 던지고, `app/global-error.tsx`가 사용자에게는 민감 정보 없이 일반 메시지만 보여주는지 확인했다

## 2. 데이터베이스 (Supabase)

- [ ] `supabase/migrations/` 아래 모든 마이그레이션이 운영 프로젝트에 순서대로 적용되었다
- [ ] Supabase 대시보드 → Database → Advisors에서 보안 경고를 확인했다 (RLS 미설정 테이블, 함수 `search_path` 등)
- [ ] `pg_cron` 작업(`generate-periodic-notifications`, 15분 주기)이 운영 프로젝트에서도 등록되어 있고 `active = true`인지 `select * from cron.job;`로 확인했다
- [ ] Storage 버킷 `attachments`가 **비공개(private)** 로 설정되어 있는지 확인했다
- [ ] 이 문서 작성 시점 기준 개발 중 생성된 테스트/더미 데이터(`scripts/create-test-users.mjs`로 만든 계정, E2E 테스트가 생성한 신청서 등)를 운영 DB에는 넣지 않는다 — 운영은 별도 프로젝트로 시작하는 것을 권장

## 3. 계정 및 권한

- [ ] 실제 경영지원팀 관리자 계정을 생성했다 (`role = 'SUPPORT_ADMIN'`, 소속 부서는 `dept_type = 'ADMIN'`)
- [ ] 각 영업국 신청자 계정을 실제 소속에 맞게 생성했다
- [ ] 개발용 테스트 계정(`sales1~3@company.co.kr`, `admin@company.co.kr`, 비밀번호 `Test1234!`)을 운영 환경에는 생성하지 않았거나, 생성했다면 비활성화(`is_active = false`)했다
- [ ] 연도별 예산(`annual_budgets`, `department_budgets`)을 실제 운영 연도 기준으로 입력했다 (`/admin/budget`)

## 4. 보안 재확인

- [ ] 로그아웃 상태에서 모든 페이지가 `/login`으로 리다이렉트된다
- [ ] 신청자 계정으로 관리자 페이지(`/admin/*`) 직접 URL 접근 시 차단된다
- [ ] 신청자 계정으로 타인/타 영업국 신청서 상세 URL 접근 시 404가 뜬다
- [ ] 첨부파일 URL을 직접 열었을 때 로그인·권한 검사를 통과한 사람만 볼 수 있다
- [ ] `.exe` 등 위장 파일(확장자만 바꾼 실행 파일) 업로드가 매직 바이트 검증으로 차단된다
- [ ] 승인 금액이 신청 금액을 초과하거나, 조정 사유 없이 금액을 바꾸는 시도가 차단된다
- [ ] 같은 신청서를 중복 승인/중복 지급 완료할 수 없다
- [ ] 브라우저 개발자 도구 콘솔·네트워크 탭에서 서버 에러 시 내부 스택 트레이스나 DB 오류 원문이 노출되지 않는다

## 5. 빌드 및 배포

- [ ] `npm run build`가 경고 없이 성공한다
- [ ] `npm run start`(프로덕션 모드)로 로컬에서 한 번 구동해, `next dev` 전용 동작(예: 상세 에러 오버레이)에 의존한 부분이 없는지 확인했다
- [ ] 배포 플랫폼의 Node.js 버전이 Next.js 15 요구사항(Node 18.18+)을 만족한다
- [ ] 배포 후 실제 도메인에서 로그인 → 신청 → 첨부 → 관리자 검토 흐름을 한 번 수동으로 확인했다

## 6. 테스트

- [ ] `npx playwright install chromium`으로 브라우저를 설치했다 (최초 1회)
- [ ] `npm run test:e2e`로 전체 흐름(신청→첨부→보완요청→재신청→승인→지급완료→예산/이력 확인) E2E 테스트가 통과한다
- [ ] E2E 테스트는 실제 DB에 데이터를 생성하므로 **운영 프로젝트에는 실행하지 않는다** (개발/스테이징 프로젝트 전용)

## 7. 운영 중 모니터링 포인트

- [ ] Supabase 대시보드에서 Storage 사용량(첨부파일 누적)을 주기적으로 확인한다
- [ ] `budget_adjustments`, `payment_corrections`, `report_downloads` 등 append-only 이력 테이블이 예상대로 쌓이는지 가끔 확인한다 (삭제/수정이 막혀 있으므로 용량은 계속 증가함 — 장기적으로는 보관 정책 검토 필요)
- [ ] pg_cron 실행 로그(`select * from cron.job_run_details order by start_time desc limit 20;`)를 주기적으로 확인해 주기 알림이 정상 동작하는지 점검한다

## 8. 알려진 제약사항 (향후 검토 과제)

이번 안정화 단계에서 발견했지만 "새 기능 추가 금지" 원칙에 따라 손대지 않은 항목입니다. 필요 시 별도로 논의해 주세요.

- 로그인/로그아웃 자체에 대한 감사 로그는 없습니다 (신청서 상태 변경, 예산 조정, 지급 처리, 리포트 다운로드 이력은 모두 기록됩니다).
- 예산 초과 승인은 시스템이 막지 않고 경고만 표시합니다(의도된 설계 — 최종 승인 권한은 관리자에게 있음). 하드 블록이 필요하면 정책 결정이 먼저 필요합니다.
- 알림은 실시간(웹소켓)이 아니라 페이지 이동 시 갱신됩니다.
- 부서 목록(영업1~3국, 경영지원팀)이 코드 여러 곳(부서 코드 `SALES1`~`SALES3`)에 하드코딩되어 있어, 조직 개편 시 마이그레이션이 필요합니다.
